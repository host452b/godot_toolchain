# 游戏开发迭代顺序

这份文档用于描述模拟经营、策略模拟、沙盒 RPG 等系统型游戏的开发顺序。核心原则是：不要一开始就做美术、剧情、音效和大内容量，先验证循环，再扩展数值、内容、反馈、UI 和引导。

适用参考样本：

- 《太阁立志传》：高自由度 RPG 沙盒与多职业成长。
- 开罗模拟经营游戏：小体量经营循环、数值反馈、目标清单。
- 《瘟疫公司》：抽象宏观模拟、风险回报、对抗式负反馈。

## 14. 如果你要自己做类似游戏，最重要的开发顺序

建议不要一开始就做美术、剧情、音效，而是先验证循环。

## 第一阶段：验证核心循环

先做最小原型：

```text
玩家操作 -> 系统反馈 -> 数值变化 -> 新选择
```

例如做经营游戏，先不需要漂亮画面，只要验证：

```text
建造 -> 顾客来 -> 赚钱 -> 升级 -> 更多顾客
```

这个循环如果不好玩，加再多美术也没用。

这一阶段重点回答：

- 玩家最小动作是什么？
- 系统是否立刻产生可理解反馈？
- 数值变化是否会带来新的选择？
- 玩家是否愿意重复这个动作闭环？
- 循环失败时，是因为操作无聊、反馈不清楚，还是奖励没有意义？

阶段验收标准：

- 可以在很粗糙的原型里连续玩 5-10 分钟。
- 玩家能不用解释就理解自己在做什么。
- 每轮操作之后都会出现新信息、新资源、新压力或新目标。

## 第二阶段：建立数值模型

做表格，先把最小经济闭环写清楚：

| 项目 | 数值 |
|---|---:|
| 初始金币 | 1000 |
| 建筑成本 | 300 |
| 每个顾客收入 | 20 |
| 顾客生成间隔 | 5 秒 |
| 升级成本 | 500 |
| 升级收益提升 | 25% |

然后测试：

- 1 分钟玩家能做什么？
- 5 分钟玩家解锁什么？
- 30 分钟是否进入重复？
- 后期是否通胀？
- 有没有失败风险？

这一阶段重点回答：

- 资源从哪里产出？
- 资源在哪里汇聚？
- 资源被什么系统消耗？
- 玩家是否永远差一点就能升级？
- 正反馈是否会滚雪球到失控？
- 负反馈是否会让玩家觉得被系统惩罚？

阶段验收标准：

- 有最小 Source / Sink 表。
- 有 1 分钟、5 分钟、30 分钟的节奏预期。
- 有至少一条成长曲线和一条难度或压力曲线。
- 可以通过改表调节节奏，而不是改代码。

## 第三阶段：做内容结构

内容不是随便堆，而是服务循环。

开罗式游戏：

```text
设施 -> 顾客 -> 员工 -> 研究 -> 评比 -> 解锁
```

太阁式游戏：

```text
人物 -> 地点 -> 技能 -> 任务 -> 事件 -> 身份
```

瘟疫公司式游戏：

```text
国家 -> 病原体 -> 传播 -> 症状 -> 解药 -> 世界事件
```

这一阶段重点回答：

- 哪些内容直接服务核心循环？
- 哪些内容只是在堆数量？
- 内容是否能复用同一套规则？
- 每个内容类型是否有明确的输入、输出和解锁条件？
- 内容扩展后，UI 和数值是否还能承受？

阶段验收标准：

- 内容类型有清晰分类。
- 每类内容都有可配置字段。
- 关键内容能通过表格、JSON、资源文件或脚本数据驱动。
- 不同内容之间能形成组合、协同或冲突。

## 第四阶段：做反馈和手感

加入：

- 点击音效
- 数字跳动
- 进度条
- 动画
- 弹窗
- 成功提示
- 失败警告
- 地图变化
- 解锁演出

这一步会极大提升“好玩感”。

这一阶段重点回答：

- 玩家每次点击是否知道系统收到了输入？
- 奖励是否有视觉、听觉或触觉反馈？
- 失败是否有明确原因，而不是只让玩家困惑？
- 重要变化是否有足够强的演出权重？
- 高频反馈是否太吵、太慢或太打断？

阶段验收标准：

- 核心动作都有即时反馈。
- 关键奖励有更强反馈。
- 风险、失败、倒计时、资源不足有可区分反馈。
- 所有反馈都服务信息传达，不只是装饰。

## 第五阶段：做 UI 和引导

尤其是模拟经营和策略游戏，UI 很可能就是游戏本体。

重点检查：

- 玩家是否知道下一步做什么？
- 重要数值是否清楚？
- 按钮是否容易找到？
- 失败原因是否明确？
- 升级收益是否可理解？
- 是否有过多弹窗打断？
- 新手是否能在 3 分钟内进入循环？

这一阶段重点回答：

- 10 秒内玩家应该关注什么？
- 1 分钟内玩家应该完成什么？
- 10 分钟内玩家应该期待什么？
- 1 小时后玩家是否还有长期目标？
- UI 是在降低认知负担，还是在展示系统复杂度？

阶段验收标准：

- 核心信息一眼可得。
- 常用操作路径足够短。
- 新手能在第一局理解核心循环。
- Tooltip、提示、目标追踪和失败解释能覆盖主要困惑点。
- 无障碍、字号、颜色、输入方式有基本考虑。

## 15. 最核心的一句话

这三类游戏虽然表面不同，但底层都是：

> 让玩家做选择，让系统产生可理解的反馈，让反馈转化为成长或压力，再推动玩家做下一次选择。

可以概括成：

```text
Decision -> Action -> Feedback -> Progression -> New Decision
```

也就是：

```text
决策 -> 行动 -> 反馈 -> 成长/变化 -> 新决策
```

做游戏时，最重要的不是先想“我要多少功能”，而是先问：

> 玩家每 10 秒、每 1 分钟、每 10 分钟、每 1 小时分别在期待什么？

## Better To Have：基础循环成立后的高级设计杠杆

下面这些不是最小原型阶段必须完成的内容。它们更适合在核心循环、数值模型、内容结构、反馈手感和 UI 引导已经基本成立后，用来提升系统深度、长期留存和玩家故事感。

## 1. 涌现式玩法 / 涌现叙事 (Emergent Gameplay / Narrative)

概念：设计师只制定简单规则，不写死剧情，由系统和 AI 在运行过程中自动碰撞出意想不到的精彩故事。

典型：《太阁立志传》。你作为一个忍者，因为在街上和某个武将喝了杯茶，结果多年后他当了城主，把你挖过去当军师。这种“老子独特的传奇人生”不是编剧写好的，而是系统涌现出来的。

设计检查：

- 是否有多个系统能互相影响，而不是各自孤立？
- NPC、势力、地点、资源、关系是否会随时间变化？
- 玩家是否能通过不同路径达成相近目标？
- 系统是否能产出“我这一局发生了独特故事”的体验？

## 2. 进度门槛 / 漏斗设计 (Progression Gating & Funneling)

概念：限制玩家在某一阶段能接触的内容，随着玩家能力提升，逐步开放新系统，防止信息过载。

典型：开罗游戏。一开始你只能建 3 种店铺，等你赚够了钱、升了级，才允许你举办活动、解锁新地块。这就是用数值和科技树做的“进度门槛”。

设计检查：

- 新玩家第一分钟是否只看到最必要的选择？
- 新系统是否在玩家已经理解旧系统之后再出现？
- 解锁是否改变玩法，而不只是增加按钮数量？
- 门槛是否让玩家期待下一步，而不是觉得被硬卡？

## 3. 动态难度调节 (DDA - Dynamic Difficulty Adjustment)

概念：游戏根据玩家当前的胜率、资源状态或顺畅度，在后台调整难度，让玩家尽量保持在心流状态。

典型：《瘟疫公司》。如果玩家前期扩散极快，突发事件、解药研发速度、国家防疫反应等参数可以动态调整，以维持紧张感。

设计检查：

- 游戏是否能识别玩家过强、过弱、卡住或失控？
- 调整是否尽量通过世界事件、AI 反应、资源供给来包装？
- 玩家是否会感觉系统在作弊？
- DDA 是否服务张力，而不是抹平玩家决策后果？

## 4. 留存机制与“再玩一回合”综合征 (The "One More Turn" Effect)

概念：核心循环的异步交叠。也就是当 A 任务结束时，B 任务刚好进行到一半，C 任务刚刚开启，导致玩家很难找到适合存档退出的空窗期。

典型：开罗游戏和《太阁立志传》。你想下班了，但太阁里主公还有 3 天就开评定会了；评定会开完，你学的新忍术还差 2 天练完；学完忍术，又刚好有人来邀请你执行任务，于是不知不觉又继续玩下去。

设计检查：

- 是否总有一个短期目标快完成？
- 是否总有一个中期目标已经投入了一半？
- 是否总有一个长期目标刚露出下一层奖励？
- 这些目标是否交叠出现，而不是同一时间全部结束？

## 5. 心流通道 (Flow Channel)

概念：玩家的技能水平 (Skill) 与游戏带来的挑战 (Challenge) 之间达到动态平衡。太难会焦虑，太易会无聊。

设计检查：

- 玩家学会一个系统后，是否很快遇到需要运用它的新挑战？
- 难度增长是否和玩家能力增长同步？
- 失败是否给出可理解原因和下一次改进方向？
- 游戏是否既有紧张段落，也有恢复、整理、扩张和规划的低压段落？

用于迭代时，可以把心流拆成四个时间尺度检查：

| 时间尺度 | 玩家期待 | 常见风险 |
|---|---|---|
| 10 秒 | 有明确输入和即时反馈 | 操作没反应、反馈太弱 |
| 1 分钟 | 有小奖励或小变化 | 目标不清、等待过长 |
| 10 分钟 | 有阶段性目标完成 | 内容重复、数值停滞 |
| 1 小时 | 有新系统、新策略或长期追求 | 后期空洞、没有新决策 |

## Reddit 社区建议整理：预生产与执行顺序

这一部分来自 Reddit 游戏开发讨论里的实践建议，重点不是“更完整的理论”，而是提醒独立开发者避免常见失败模式：先写太多故事、规划太久、迟迟不做可玩的东西。

## 1. 游戏玩法优先于故事

多数回复者一致认为：先确定游戏机制和玩法，再处理故事。

原因：

- 玩法灵活性低，故事相对容易调整。
- 过早写故事容易产生虚假进度感。
- 如果核心机制不好玩，故事越完整，后期越难舍弃。
- 玩法确定后，故事可以反过来服务玩家行为和系统反馈。

落地方式：

- 先写一句话玩法：玩家做什么，系统如何回应。
- 先做可玩原型，再写完整世界观。
- 故事只保留能解释目标、动机、反馈和主题的部分。

## 2. 预生产的核心步骤

一个较完整的预生产框架可以拆成五步：

| 阶段 | 目标 | 产物 |
|---|---|---|
| 概念阶段 | 确定游戏类型和核心循环 | Genre、Core Loop、目标玩家 |
| 系统规划 | 为每个循环动作定义所需系统 | 系统清单、依赖关系、数据输入输出 |
| 原型制作 | 从主循环开始，逐步扩展次要循环 | 可玩原型、节奏验证、失败记录 |
| 美术与测试 | 加入视觉元素并持续迭代测试 | 临时美术、可读性测试、体验反馈 |
| 最终打磨 | 优化图形、音效、反馈和整体品质 | Polish 清单、音效/VFX/UI 调整 |

关键点：

- 主循环先于次要循环。
- 系统清单要从玩家动作倒推，而不是从想做的功能堆出来。
- 美术进入得太早会掩盖玩法问题。
- 打磨不是补救核心循环，而是放大已经成立的体验。

## 3. 游戏开发是迭代过程

游戏开发不是线性地“完成一项再做下一项”，而是不断回顾、修改、扩展。

更现实的流程是：

```text
假设 -> 原型 -> 测试 -> 发现问题 -> 调整 -> 再测试
```

迭代时重点看：

- 玩家是否理解目标？
- 玩家是否自然做出你期待的行为？
- 玩家是否在错误时间感到无聊、困惑或挫败？
- 新系统是否真的改善体验，还是只增加复杂度？

## 4. 用垂直切片估算工期

垂直切片是先做一个关卡、场景或玩法段落的完整版本，用它推算整体工作量。

它应该包含：

- 核心玩法
- 基础 UI
- 基础美术
- 基础音效
- 关键反馈
- 最小内容链路
- 一次完整开始、推进、结束体验

用途：

- 估算真实制作成本。
- 发现隐藏依赖和技术债。
- 判断当前品质目标是否过高。
- 为后续内容生产建立模板。

注意事项：

- 用垂直切片估算全项目时，要预留范围蔓延和返工缓冲。
- 如果一个小切片都做不完，完整项目范围大概率过大。

## 5. 避免过度规划的陷阱

常见失败模式：

- 规划过度，执行不足。
- 第一个项目范围过大。
- 路线图写得很细，但没有任何可玩验证。
- 把 GDD 当进度，而不是把可玩原型当进度。

更稳妥的做法：

- 第一个项目保持极简。
- 尽快动手做最小可玩版本。
- 把路线图视为预生产阶段的猜测。
- 每次测试后都允许路线图调整。
- 不要因为已经写了设定、剧情或系统文档，就拒绝砍范围。

## 6. 实用工具建议

可用工具：

- 游戏设计文档 (GDD)：记录核心支柱、循环、系统、数值、内容结构。
- 思维导图：用 XMind 之类工具做全局概览。
- 简单电子表格：管理任务优先级，如 A1 / A2 / B1。
- 甘特图：梳理依赖关系和关键路径。

使用原则：

- 工具服务决策，不服务形式感。
- GDD 要跟随原型测试更新。
- 优先级表要能指导今天做什么。
- 甘特图只在依赖复杂时才值得维护。

## 7. 尽早启动营销

很多独立游戏建议在概念阶段就开始最小营销准备。

可做事项：

- 建立 Steam 页面。
- 注册社交媒体账号。
- 做简易官网或项目页。
- 定期发布原型进展。
- 收集愿望单、关注者和早期反馈。

原因：

- 游戏上线当天才开始营销通常太晚。
- 早期页面能验证题材吸引力。
- 社区反馈能帮助发现定位问题。
- 愿望单和关注数据能反过来影响范围判断。

注意事项：

- 不要为了营销承诺尚未验证的功能。
- 不要让营销素材倒逼你保留不好玩的系统。
- 早期展示应聚焦核心卖点和真实可玩内容。
