# 玩法与游戏设计专业术语表

这是一份面向游戏设计、系统拆解、Godot 项目规划和竞品分析的人类可读术语表。它不是 API 文档，也不是实现规范；它的作用是统一讨论语言，避免把“玩法”“数值”“反馈”“工程结构”混在一起说。

示例样本：

- 《太阁立志传》：高自由度 RPG 模拟，重点是身份、职业、事件、关系和历史沙盒。
- 开罗模拟经营游戏：Checklist 经营，重点是资源投入、目标可见性、检验包装和高频反馈。
- 《Plague Inc.》：系统对抗策略，重点是传播模型、全局 UI、反制系统和数值张力。

## 三款样本的核心拆解

| 样本 | 最小核心循环 | 关键设计判断 |
|---|---|---|
| 《Plague Inc.》 | 感染 -> 攒 DNA 点数 -> 进化传播/症状/抗性 -> 世界反应变化 -> 再调整。 | 核心循环极其干净，一句话能讲清；张力来自“扩散越猛，反制越强”。 |
| 开罗《游戏发展国》类 | 立项做产品 -> 卖出赚钱/涨粉 -> 升级研发/招人 -> 做更大的产品。 | 数值循环和检验包装是核心；玩家一直在变强，但永远差一点。 |
| 《太阁立志传》 | 没有单一核心循环，而是武士、忍者、商人、医师、茶人等多套职业循环并行。 | 自由度来自多循环并行；成本是系统宽度、内容量、UI 复杂度和新手门槛都很高。 |

### 循环的分层

| 层级 | 时间尺度 | 示例 |
|---|---|---|
| 微观循环 | Moment-to-moment，几秒 | 瘟疫里点一次进化；开罗里做一个开发动作；太阁里拜访、移动、修业一次。 |
| 中观循环 | Session，一局、一个产品周期、一年 | 瘟疫一局疫情；开罗一个产品从立项到发售；太阁一次主命或一段职业目标。 |
| 宏观 / Meta 循环 | 跨局、跨周目、长期生涯 | 瘟疫解锁新病原体；开罗新游戏+继承；太阁统一天下、极めた、全职业/结局目标。 |

## 设计层核心术语

| 术语 | 定义 | 三款游戏中的体现 |
|---|---|---|
| Core Loop / Core Gameplay Loop | 贯穿全生命周期、最高频的玩家行为闭环。 | 太阁：巡回 -> 任务 -> 晋升；开罗：等待 -> 投入 -> 检验；瘟疫：传播 -> 进化 -> 应对。 |
| Meta Loop / Outer Loop | 局外或长期成长系统，例如存档继承、解锁、收藏、难度、结局。 | 太阁：通关解锁新职业、结局、难度；开罗：新游戏+、图鉴/博物馆收藏；瘟疫：基因解锁、作弊模式。 |
| Resource Loop / Economy Loop | 资源来源 -> 汇聚 -> 消耗 -> 再投资的内部经济闭环。 | 太阁：体力/金钱 -> 功勋 -> 身份 -> 更高收益；开罗：金币 -> 建筑 -> 效率 -> 金币；瘟疫：DNA -> 特性 -> 感染 -> DNA。 |
| Compulsion Loop / 成瘾循环 | 行为 -> 奖励 -> 期待 -> 再行为的心理机制。 | 开罗最典型：高频不确定奖励；瘟疫：DNA 气泡收集；太阁：卡片、稀有物品、事件收集。 |
| Goal Visibility / 目标可见性 | 短期目标强提示，长期目标弱提醒，让玩家始终知道下一步。 | 开罗：任务栏常驻；太阁：主命面板；瘟疫：解药进度条、感染国家计数。 |
| Verification Phase / 检验阶段 | 操作后延迟揭晓结果，用结算包装不确定性和期待。 | 开罗：建筑建成、比赛结算；瘟疫：国家感染、解药突破；太阁：合战、茶道、考试结果。 |
| Goal Gradient Effect / 目标梯度效应 | 越接近目标，动机越强。 | 三款均利用：晋升在即、建筑即将满级、全球感染倒计时。 |
| Numerical Loop / 数值循环 | 成本、收益、成长、消耗之间的数值闭环。 | 开罗偏再投资膨胀；瘟疫偏对抗曲线；太阁偏门槛解锁。 |
| Source / Sink / 产出与回收 | Source 是资源产出口，Sink 是资源消耗口。 | 金币、DNA、功勋都是 Source；建筑、特性、身份推进都是 Sink。 |
| Progression Curve / 成长曲线 | 玩家能力、资源、内容、复杂度随时间变化的曲线，可线性、指数、对数或分段。 | 开罗用指数膨胀配合同步成本膨胀；瘟疫压力逐步升级；太阁由身份和技能拉长。 |
| Inflation / 数值膨胀 | 数值增长过快，导致早期成本、奖励或挑战失去意义；也可以被刻意用于爽感。 | 开罗刻意让数字滚大制造爽感；瘟疫刻意压制 DNA 让资源长期紧张。 |
| Combo / Synergy / 组合协同 | 多个元素组合后产生额外收益或策略分支。 | 开罗的类型搭配、设施相性和彩蛋加成，是经济深度的重要来源。 |
| Positive / Negative Feedback / 正负反馈 | 正反馈放大成功，负反馈抑制领先或扶持落后。 | 开罗收入滚雪球；瘟疫感染越多 DNA 越多，同时解药形成负反馈。 |
| Snowball Effect / 滚雪球 | 早期优势持续放大，压倒其它策略。 | 开罗设施组合、瘟疫早期扩散、太阁早期身份优势都可能滚雪球。 |
| Risk / Reward / 风险回报 | 玩家承担风险以换取更高收益。 | 瘟疫高致命性带来更快死亡但更高警觉；太阁高风险任务换功勋。 |
| Flow / 心流 | 玩家技能与挑战动态匹配时产生的投入状态。 | 开罗靠密集反馈，瘟疫靠压力曲线，太阁靠自由目标和成长节奏。 |
| Difficulty Curve / 难度曲线 | 挑战强度随时间、关卡或玩家能力变化的曲线。 | 瘟疫最明显：解药、国家防控、病原体类型共同形成曲线。 |
| Dynamic Difficulty Adjustment, DDA / 动态难度 | 根据玩家表现动态调整挑战、资源或压力。 | 可用于瘟疫的事件概率、开罗的评比阈值、太阁 NPC/势力压力。 |
| Pacing / 节奏 | 奖励、压力、学习、休息、高潮的安排。 | 太阁在评定、修业、旅行间切换；开罗用建造和结算交错；瘟疫用爆发和反制交错。 |
| Player Agency / 玩家能动性 | 玩家选择对世界、系统和结果的实际影响程度。 | 太阁最高，玩家路线改变身份和局势；瘟疫是宏观调参能动性；开罗是布局和经营选择。 |
| Emergent Gameplay / 涌现式玩法 | 简单规则组合后产生非脚本化结果。 | 太阁最典型：NPC、势力、事件、身份系统碰撞出个人故事。 |
| System-driven / 系统驱动 | 乐趣主要来自规则、模拟、AI、经济和状态变化。 | 瘟疫和太阁更偏系统驱动。 |
| Content-driven / 内容驱动 | 乐趣主要来自关卡、剧情、角色、任务、美术资产等手工内容。 | 太阁的历史事件和人物池、开罗的设施主题都需要内容驱动支撑。 |
| MDA Framework | Mechanics 机制 -> Dynamics 动态 -> Aesthetics 体验的分析框架。 | 太阁：卡片/属性 -> 自由叙事 -> 角色扮演和成就感。 |
| Core Pillars / 核心支柱 | 游戏设计不可妥协的 3-5 条体验承诺。 | 太阁：自由度、历史沉浸、养成深度；开罗：轻松、收集、成长；瘟疫：真实感、策略深度、压迫感。 |
| Player Fantasy / 玩家幻想 | 玩家希望自己在游戏中成为谁、拥有何种力量或处境。 | 太阁是战国人生；开罗是经营大师；瘟疫是操控全球系统的反常规幻想。 |
| Power Fantasy / 代入感 / 权力幻想 | 玩家在系统中获得能力、身份、控制力或社会地位的体验。 | 太阁偏身份和人生代入；开罗偏经营掌控；瘟疫偏宏观操控。 |
| Ludonarrative Consistency / 玩法叙事一致性 | 游戏规则与叙事表达互相支持。反面是 ludonarrative dissonance。 | 太阁身份成长应支撑历史沉浸；瘟疫 UI 与音频应支撑全球危机叙事。 |
| Environmental Storytelling / 环境叙事 | 通过空间、道具、布局、痕迹、声音和视觉状态讲故事。 | 太阁城镇和历史地点；开罗设施布局和小人行为；瘟疫地图颜色和新闻变化。 |

## 系统 / 技术层术语

| 术语 | 说明 |
|---|---|
| Game Loop 技术层 | 经典结构是 `while (running) { processInput(); update(dt); render(); }`，重点是固定时间步长、帧率无关更新和输入/模拟/渲染分离。 |
| Finite State Machine, FSM / 有限状态机 | 状态 + 事件 + 转移；用于 AI、UI 流程、技能、关卡阶段。 |
| Hierarchical FSM, HFSM / 层次状态机 | 状态嵌套，例如 `战斗 -> 近战/远程 -> 攻击/格挡`。适合复杂角色或系统阶段。 |
| Gameplay FSM / 游戏逻辑状态机 | 控制疾病阶段、NPC 行为、对话流程、任务阶段、国家反应等逻辑状态。 |
| Animation State Machine / 动画状态机 | 控制待机、走、跑、攻击、格挡、受击等动画切换，是动作手感的底层之一。 |
| Behavior Tree / 行为树 | 比 FSM 更易组合复杂 AI，例如太阁 NPC 外交、合战、拜访和修业决策。 |
| GOAP / Goal-Oriented Action Planning | AI 根据目标和可用行动自动规划行为序列。适合目标多、行动可组合的 NPC。 |
| Data-driven Design / 数据驱动 | 数值和配置外置到 Excel、CSV、JSON、Godot Resource、Unity ScriptableObject 等，策划无需改代码即可调平衡。 |
| Event Bus / 事件总线 | 解耦系统间通信，例如 `infection_spread` 事件触发 UI 刷新、音效、成就和存档标记。 |
| Entity Component System, ECS | 用实体、组件、系统组织大量对象，适合高性能大量实体处理；不一定每个游戏都需要。 |
| ScriptableObject / 数据容器 | Unity 常用的数据资产化方式；在 Godot 中通常可类比为自定义 `Resource`。 |
| Save/Load Architecture | 存档架构，包括快照、增量、版本迁移。例：太阁多结局存档、开罗离线收益、瘟疫断局续玩。 |
| Tick Simulation / Tick 模拟 | 系统按固定步长更新，例如每帧、每物理帧、每游戏日、每经营周期。 |
| Determinism / 确定性 | 相同输入与初始状态总能得到相同结果。用于回放、同步、测试和模拟复现。 |
| RNG / Seed / 随机数与种子 | 随机事件、掉落、生成和模拟扰动的来源；固定 seed 可用于复现和调试。 |
| Config Table / 配置表 | 存放建筑、病原体、事件、NPC、奖励、解锁等参数的数据表。 |
| Serialization / 序列化 | 把运行时状态转换成可保存、可传输、可恢复的数据。 |
| Procedural Generation / 程序化生成 | 用规则或算法生成地图、任务、事件、病原体变异或经营布局。 |
| Tunability / 可调参 | 系统参数易于查找、修改、测试和回滚。关键是命名、范围、默认值和验证工具。 |

## 交互 / 表现层术语

| 术语 | 说明 |
|---|---|
| Game Feel / Juice / 手感 / 游戏汁液 | 输入响应、动画缓动、粒子、屏幕震动、音效、时间缩放和数字反馈的综合体验。 |
| Input Latency / 输入延迟 | 从按键、点击或触摸到屏幕反馈的时间。移动端常见目标是低于 100ms。 |
| Hit Stop / 顿帧 | 命中或冲击瞬间短暂停顿，用于强化重量感和打击反馈。 |
| Easing / 缓动 | 动画变化的速度曲线，例如 ease-in、ease-out、bounce。 |
| Impact Feel / 打击感 | 命中反馈的综合质量，包括时机、音效、动画、停顿、位移和特效。 |
| Adaptive Audio / 自适应音频 | 音乐或音效根据游戏状态动态变化，例如危机、胜利、低血量、爆发、解药突破。 |
| SFX / Sound Effect / 音效 | 操作确认、奖励、风险提示、状态变化和情绪塑造的短音频反馈。 |
| BGM / Background Music / 背景音乐 | 塑造场景、模式和压力状态的持续音乐层。 |
| Feedback Triad / 反馈三要素 | 常用可读反馈组合：视觉、听觉、触觉/交互响应。关键信息最好不依赖单一通道。 |
| HUD / Heads-Up Display | 非模态常驻信息层，例如资源、时间、小地图、解药进度、任务目标。 |
| UI/UX / 用户界面与体验 | 信息架构、导航流、可访问性、触屏/手柄/键鼠适配、字号和交互成本。 |
| Affordance / 可供性 | 元素通过形状、状态、动效或位置暗示“可以被操作”。 |
| Information Hierarchy / 信息层级 | 信息的重要性排序和视觉组织。决定什么最显眼、什么可折叠。 |
| Telegraph / 预兆 | 危险、攻击、事件或变化前给出的可读提示。 |
| First-Time User Experience, FTUE / 新手引导 | 玩家第一次进入游戏时的学习和上手体验。目标是快速进入核心循环。 |
| Feedback Loop / 反馈回路 | 玩家行动和系统回应形成的闭环；正反馈推动成长，负反馈形成追赶、衰减或反制。 |
| Accessibility / 无障碍 | 重映射、单手模式、高对比度、文本转语音、字号、字幕、难度可调、色盲友好等。 |

### 不同类型里的手感差异

| 类型 | 手感来源 |
|---|---|
| 动作游戏 | 打击感、顿帧、屏幕震动、粒子、缓动、输入延迟、动画取消窗口。 |
| 模拟经营 | 数字反馈爽度、弹字、进度条填满、升级音效、结算节奏、小人忙碌感。 |
| 宏观策略 | 地图变化、颜色扩散、全局计数器、预警音效、系统失控的可视化。 |

瘟疫公司的 UI/UX 是宏观策略范本：感染数、死亡数、治疗进度要一眼可得；地图颜色和新闻系统承担了大量系统解释。太阁则是反面教训之一：系统极深导致信息过载，新手劝退是宽系统的固有成本。

## 运营 / 扩展层术语

| 术语 | 说明 |
|---|---|
| LiveOps / 活动运营 | 限时活动、赛季通行证、每日任务、远程配置、运营节奏和回流设计。 |
| Modding Support / Mod 架构 | 数据外置、脚本接口、Hook、热更新、Steam Workshop/API 等扩展能力。 |
| Procedural Generation / 程序化生成 | 在运营层常用于事件池、地图、挑战、病原体变异、经营主题和关卡组合。 |
| Telemetry / 埋点分析 | 漏斗、留存、数值分布、卡关点、经济积压、Mod 订阅量等数据采集。 |
| Balance Metrics / 平衡指标 | 胜率、资源积压率、关卡通过率、DNA 点获取曲线、建筑回本时间、任务完成率等。 |
| Replayability / 重复可玩性 | 玩家多次游玩仍有新路线、新组合、新挑战或新目标。 |
| Roguelike / Roguelite | 通过随机生成、永久死亡、局外成长等提高重复可玩性的类型结构。 |
| Permadeath / 永久死亡 | 失败后角色或本局进度永久丢失，强化风险和重开价值。 |
| Meta-progression / Meta 进度 | 跨局保留的成长、解锁、收藏或账号进度。 |
| Retention / 留存 | D1/D7/D30 等指标衡量玩家是否回来继续玩。 |
| Content Consumption / 内容消耗速度 | 玩家消耗内容的速度；消耗快于生产会进入“长草期”。 |
| Monetization / 变现模型 | 买断、广告、内购、DLC、订阅、通行证等商业化结构。 |

Mod 能力本质上是数据驱动能力的延伸。一个游戏能不能 Mod，取决于它是否把角色、事件、数值、地图、规则和脚本接口从硬编码中分离出来。

## 制作 / 职能向

| 术语 | 英文 | 简要定义 | 制作时要问 |
|---|---|---|---|
| 平衡性 | Balancing | 调整数值、成本、奖励、难度和策略空间，使体验符合目标。 | 是真的不平衡，还是反馈让玩家误判？ |
| 数值策划 | Economy / Balance Designer | 负责资源、成长、公式、奖励、难度和长期曲线。 | 是否有表格、模拟和验证指标？ |
| 系统策划 | Systems Designer | 负责玩法系统规则、交互、状态、依赖和可扩展性。 | 规则是否清晰到程序可以直接实现？ |
| 关卡策划 | Level Designer | 负责空间、路径、节奏、遭遇、资源摆放和玩家引导。 | 空间是否服务玩法，而不是只服务美术？ |
| 玩法测试 | Playtest | 通过观察玩家验证玩法、理解、难度、节奏和反馈。 | 记录的是玩家行为，还是只听玩家评价？ |
| 迭代 | Iteration | 基于测试结果持续调整设计、实现和内容。 | 每轮改动是否有明确假设和验证目标？ |
| 技术债 | Technical Debt | 为了速度或不确定性留下的工程负担。 | 债务是否被记录、分级，并有偿还条件？ |

## 三类游戏的第一优先级对比

| 优先级 | 太阁型：高自由度 RPG 模拟 | 开罗型：Checklist 经营 | 瘟疫型：系统对抗策略 |
|---|---|---|---|
| P0 | 身份/职业状态机、主命-功勋-晋升链、内容池：事件、卡片、武将。 | 资源产出/消耗公式、目标可见性、检验包装、高频正反馈。 | 传播模型参数、全局信息 UI、反制系统、数值张力曲线。 |
| P1 | 地图巡回节奏、技能/卡片成长树、迷你游戏多样性。 | 建筑解锁顺序、离线收益公式、收藏图鉴。 | 病原体差异化特性、难度曲线、遗传密码/基因解锁。 |
| P2 | 战斗/外交 AI、美术风格统一、存档/继承系统。 | 美术风格统一、音效 Juice、本地化。 | 世界事件/新闻系统、成就/统计面板、Mod 编辑器。 |

## 三款样本适合学习的环节

| 想学习的能力 | 优先拆解对象 | 原因 |
|---|---|---|
| 数值循环、经济、上瘾节奏、模拟器手感 | 开罗，尤其《游戏发展国》 | 它把 Source/Sink、成长曲线、目标可见性、检验包装和正反馈做得非常紧凑。 |
| 张力曲线、对抗式反馈、风险回报、UI 可读性、清晰核心循环 | 《Plague Inc.》 | 它的核心循环干净，负反馈强，UI 几乎就是玩法本体。 |
| 系统宽度、涌现式玩法、自由度、多循环并行、NPC 模拟 | 《太阁立志传 V》 | 它是系统驱动 + 开放目标 + 多职业循环并行的典型高难样本。 |

## 快速分析模板

分析一个系统时，可以按这个顺序走：

```text
1. Player Fantasy: 玩家幻想是什么？
2. Core Pillars: 哪些体验承诺不可妥协？
3. Core Loop: 玩家最高频行为闭环是什么？
4. Resource Loop: 资源如何产出、汇聚、消耗、再投资？
5. Goal Visibility: 玩家是否知道短期和长期目标？
6. Verification Phase: 操作结果如何被检验和揭晓？
7. Feedback: 玩家如何知道自己做对或做错？
8. Progression: 这个循环如何产生长期变化？
9. Risk: 滚雪球、最优解、数值膨胀、信息过载在哪里？
10. Implementation: 需要 FSM、事件系统、Tick、数据表还是存档？
11. Playtest: 用什么可观察行为证明它成立？
```

## 深挖练习建议

- 开罗式经营：还原 `金币 -> 建筑 -> 顾客 -> 收入 -> 升级 -> 解锁` 的数值表，重点看 payback time、Source/Sink、Combo、Goal Visibility 和数值膨胀。
- Plague Inc. 式反馈系统：还原 `传播 -> DNA -> 进化 -> 警觉 -> 解药 -> 反制`，重点看正反馈、负反馈、地图可视化、压力曲线和 Verification Phase。
- 太阁式沙盒：还原 `身份 -> 任务 -> 技能 -> 关系 -> 事件 -> 世界局势`，重点看玩家能动性、涌现叙事、事件触发、数据驱动和 HFSM。
